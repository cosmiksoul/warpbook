# CLAUDE.md — quackbook

Правила для Claude Code на этом проекте. Источник истины по скоупу — `docs/scope-quackbook-v1.md`. Если что-то здесь противоречит скоупу — побеждает скоуп, и сообщи о расхождении.

## Что строим

Браузерный аналитический ноутбук на DuckDB-WASM. Два слоя: **данные** (drop CSV/Parquet → схема в рейле → SQL с join/union → Профиль значений) и **ноутбук** (закрепляешь результат виджетом → вписываешь текст → собираешь нарративный отчёт). Всё в браузере, без бэкенда, статика на GitHub Pages. v1 — только локальные файлы. Полный скоуп — в `docs/scope-quackbook-v1.md`.

## Стек и жёсткие ограничения

- **Движок:** DuckDB-WASM 1.5.4 (Web Worker, Arrow) — это версия **движка** DuckDB; npm-пакет `@duckdb/duckdb-wasm` запинен на `1.32.0` (разные схемы версий — не путать и не «чинить»). Не тащим серверный DuckDB, не делаем бэкенд.
- **Сборка/деплой:** Vite → GitHub Pages. Только статика.
- **Данные:** только локально загруженные файлы, всё исполнение на клиенте. Никаких сетевых запросов к данным в v1 (remote Parquet вырезан осознанно — CORS + хрупкий range-детект).
- **Каркас (vanilla vs React): НЕ выбран. Это первое решение — прими его явно, зафиксируй причину, дальше не смешивай подходы.**
- **Редактор:** CodeMirror 6. **Markdown:** marked. **Грид:** виртуализированный. **Виз:** Observable Plot / Chart.js (Recharts — если React).

## Базовый процесс: TDD

Для всего, что содержит логику (построение SQL, обработка инференса схемы, каст-логика `TRY_CAST`, сериализация блоков отчёта, key-хинт, профиль) — **сначала тест, потом код**:

```
1. Красный  — тест на желаемое поведение (падает).
2. Зелёный  — минимальный код, чтобы прошёл.
3. Рефактор — чистим, тесты остаются зелёными.
```

Vitest. Задачи переводим в проверяемые цели: «добавь валидацию» → «тест на невалидный ввод, потом зелёный»; «почини баг» → «тест, воспроизводящий баг, потом зелёный».

**Честная граница:** чистую презентацию (CSS, раскладка, цвета) тестами не покрываем — проверяется глазами. TDD — для данных и логики, где он ловит регрессии, а не догма на каждый пиксель.

## Правила

1. **Думай прежде чем действовать.** Перед нетривиальной задачей — сформулируй понимание и допущения вслух. Запрос неоднозначен → стоп, скажи что неясно, спроси. Не выбирай интерпретацию молча.

2. **Простота вперёд.** Минимум кода, решающий задачу. Никаких абстракций «на будущее» без запроса. Никакого error handling под несуществующие сценарии. 200 строк там, где хватает 50 → перепиши.

3. **Хирургические правки.** Трогай только то, что нужно. Не «улучшай» соседний код, форматирование, комментарии. Не рефактори то, что не сломано. Видишь рядом баг/мусор — **скажи**, не правь молча.

4. **Stop and replan.** Дважды подряд не выходит одно и то же (тест не зелёный, баг вернулся) — стоп. Опиши: что пытался, почему не выходит, возможную архитектурную причину, что предлагаешь. Жди решения. Не костыляй.

5. **Trace-ability.** Каждая правка трассируется к скоупу (`docs/scope-quackbook-v1.md`) или к прямому запросу. Не можешь сослаться — возможно, додумал. Скажи об этом.

6. **Уважение к firewall.** Скоуп явно перечисляет, что вне v1 (ниже). Не предлагай это «потому что так проще». Кажется, что граница неверна — **обсуди явно**, не обходи молча.

7. **Открытость к коррекции.** «Ты не учёл X» — ценная информация, не атака. Прими, скорректируй, продолжай. Без длинных оправданий.

## Firewall — НЕ строить без явного запроса

Главный риск проекта — scope creep. Ловишь себя на одном из этого — **стоп, это вне v1, спроси**:

- **Dashboard-грид / канвас / drag-resize плитки / много колонок.** Отчёт — вертикальный стек, ширина одна. Перестановка по вертикали — да, раскладка — нет. Это линия, отделяющая нас от Superset.
- **Визуальный join-builder** (перетащи колонку на колонку). JOIN/UNION только через SQL + key-хинт.
- **Правка значений** (per-cell), деривативные/вычисляемые колонки в редакторе схемы. Путь в недо-Excel.
- **Персист данных** (OPFS), мультипользовательность, шеринг с правами — позже.
- **Таб «План»** (EXPLAIN ANALYZE) — не сейчас.

- `npm test` — Vitest (node-окружение)
- `npm run build` — продакшн-сборка в `dist/` (полный type-check)
- `npm run preview` — предпросмотр сборки
- `npm run lint` — ESLint
- **Деплой:** push в `main` → GitHub Actions публикует на Pages.
