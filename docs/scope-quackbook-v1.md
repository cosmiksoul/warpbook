# quackbook — Scope (v1, notebook-направление)

> Рабочее название отсылает к DuckDB и ноутбук-лайк софтам (Jupyter, Colab). Продукт — не скретчпад и не дашборд, а **растущий ноутбук**, который из CSV собирает читаемый отчёт-нарратив. Движок (DuckDB-WASM), схема в рейле и Профиль — ядро; ноутбук — верхний слой над ними.

## Референс

Если смешать три инструмента: **BigQuery** (исследование данных, схема, профиль значений), **Colab / Jupyter** (ноутбук-нарратив из ячеек) и **Sublime** (быстрый редактор с табами). quackbook берёт исследовательскую мощь первого, повествовательный формат второго и табовую лёгкость третьего.

## Суть

Браузерный аналитический ноутбук. Пользователь бросает CSV/Parquet, исследует данные (схема всегда перед глазами, Профиль показывает, что внутри колонок), пишет SQL — в том числе джойны и юнионы между несколькими файлами, — **закрепляет результат блоком-виджетом** (сохранённые запрос + визуализация) и **вписывает текстовые комментарии между блоками**, собирая отчёт, который читается сверху вниз как рассуждение. Всё на клиенте, в WebAssembly, без бэкенда. Деплой — статика (GitHub Pages). v1 работает только с локально загруженными файлами.

## Зачем

Технодемка для портфолио с честным «зачем», выросшим из реальных болей аналитика в BigQuery:

1. **Схема не перед глазами**, когда пишешь запрос. → Здесь схема всегда в левом рейле.
2. **Непонятно, какие значения в данных.** → Здесь Профиль показывает distinct-values и распределения одним проходом.
3. **Воспроизводимость результатов и их перенос в отчёт.** → Здесь отрабатывает слой ноутбука: каждый виджет несёт свой SQL и имя датасета.

Сверху — формат, превращающий набор чартов в аргумент: **ноутбук-нарратив**. Инструмент на выходе даёт артефакт (отчёт), а не просто гоняет запросы — в портфолио это читается как «человек умеет рассказать данными», что ценнее десятка дашбордов.

Сознательно **не** «супер-лёгкий Superset»: цель формата — повествование (Observable / Colab notebook), а не пространственный канвас с плитками. Это удерживает центр тяжести проекта на движке данных (твой edge), а не на layout-плумбинге (generic-фронтенд).

## Демо-нарратив (что зритель видит)

1. Бросает CSV (можно несколько) → схема в рейле, Профиль показывает, что в колонках.
2. Кликает датасет → открывается таб исследования. Пишет запрос — при желании джойнит/юнионит несколько файлов.
3. Смотрит результат (таблица / график).
4. **Закрепляет** результат блоком в ноутбук.
5. Вписывает текстовый блок-комментарий: что заметил.
6. Повторяет 2–5 ещё пару раз.
7. На выходе — вертикальный отчёт «данные → наблюдение → срез, подтверждающий наблюдение». Экспортирует в PDF/HTML и пересылает.

## Ядро: два слоя

**Слой данных** (ядро, твой edge):
`IDLE → PROBE → EDIT_SCHEMA → MATERIALIZE → QUERY` — sniff + безопасный baseline + `TRY_CAST` со счётчиком, материализация, SQL в Web Worker, результат в Arrow. Схема в рейле с pruning-подсветкой. Профиль (distinct-values + гистограммы) одним проходом.

**Слой ноутбука** (верхний, новый):
Документ из блоков двух типов:
- **Текстовый блок** — markdown-комментарий, связующая ткань нарратива.
- **Виджет-блок** — закреплённый {запрос + визуализация}. При открытии отчёта перезапускается по таблицам в памяти.

Блоки складываются в вертикальный стек, переставляются вверх-вниз. **Ширина всегда одна.**

**Анатомия виджет-блока:** заголовок · имя датасета(ов)-источника · сворачиваемый SQL-сниппет, который был закреплён · визуализация · подпись. SQL и имя датасета — это и есть воспроизводимость (боль №3): по блоку видно, *что* за запрос и *по каким* данным он построен.

## Навигация и сессия

**Табы (Sublime / BQ-style).** Постоянный таб **«Отчёт»** + табы **исследования**: клик по загруженному датасету в рейле открывает (или фокусирует) таб исследования, преднастроенный на этот датасет (`SELECT * FROM <table>`).

**Важно — табы не изолируют данные.** Все загруженные файлы живут в одной in-memory DuckDB, поэтому запрос из любого таба видит все таблицы. Отсюда прямой ответ на «можно ли джойнить/юнионить несколько CSV из редактора»: **да, это нативно** — каждый файл это таблица, `JOIN` и `UNION` (включая удобный `UNION ALL BY NAME` для стопки CSV с разным порядком колонок) пишутся обычным SQL, без какого-либо плумбинга. Таб от датасета — лишь удобная точка старта, не песочница.

**Сессия.** Пока вкладка браузера открыта, стейт сессии (загруженные таблицы, табы, отчёт) держится в памяти и переживает переключения режимов/табов и прогоны запросов. Кнопка **Reset** — явная очистка сессии (drop таблиц, пустой отчёт и табы).

**Перезагрузка страницы — честная оговорка.** In-memory таблицы при reload теряются (ограничение WASM). v1-поведение: при открытии сохранённого отчёта просим заново подгрузить исходные датасеты, после чего код перезапускается. Виджет знает имя своего датасета, поэтому подсказывает, какой файл нужен. Сквозной персист данных между сессиями — OPFS, в v1.5.

## В скоупе v1

**Слой данных:**
- Drag-and-drop локальных CSV и Parquet (несколько файлов).
- CSV: инференс (`sniff_csv`) + `all_varchar` baseline + one-click типы.
- Правки схемы: rename, тип, формат даты/decimal, nullstr, include/exclude.
- `TRY_CAST` + счётчик «N → NULL».
- Несколько файлов → несколько таблиц; **`JOIN` и `UNION` между файлами обычным SQL** (нативно для DuckDB, без плумбинга).
- **Key-хинт от Профиля**: когда у двух таблиц колонки совпадают по имени/типу/кардинальности, подсказка «events.user_id ↔ users.id, сджойнить?» вставляет `JOIN` в SQL одним кликом. Подсказка, не конструктор. *(вырезан полностью решением — см. DECISIONS.md, 2026-06-22; ни v1, ни v1.5)*
- Схема всегда в рейле + pruning-подсветка используемых колонок.
- Профиль: на `SUMMARIZE` (min/median/max/null%/uniq) + точечные `GROUP BY` для топ-значений категориальных. Бары для категориальных, гистограмма для числовых.

**Навигация:**
- Табы: постоянный таб «Отчёт» + табы исследования, открываются по клику на датасет.
- Все табы делят одну in-memory DuckDB (запрос видит все загруженные таблицы).
- Кнопка **Reset** — очистка сессии.

**Слой ноутбука:**
- Два типа блоков: текст (markdown) и виджет (запрос + виз).
- Закрепление результата запроса виджетом в ноутбук.
- **Виджет показывает сохранённый SQL + имя датасета-источника** (воспроизводимость).
- **Чистое разделение Таблица / График**: таблица — плотная сетка точных значений (без баров в ячейках), график — визуальное кодирование.
- Вертикальная перестановка блоков (drag по вертикали).
- Сохранение/загрузка **структуры** отчёта (текст + запросы + виз) — без данных.
- **Экспорт отчёта в HTML и PDF** (self-contained HTML; PDF через print-CSS / `window.print` или lib типа html2pdf).

## Вне скоупа v1 (firewall)

- **Dashboard-грид / freeform-канвас / drag-resize плитки / многоколоночная раскладка.** Граница формата: блоки переставляются по вертикали, ширина всегда одна. Перестановка — да, раскладка — нет. Это и есть линия, отделяющая ноутбук от Superset.
- **Визуальный join-builder** (перетащи колонку на колонку) — JOIN/UNION только через SQL + key-хинт.
- **Персистентность данных** (OPFS) — v1 сохраняет только структуру, файл подгружается заново при открытии (см. «Фазы»).
- Правка значений (per-cell), деривативные колонки в редакторе схемы.
- Многопользовательский режим, шеринг с правами, коллаборация.
- Таб «План» (EXPLAIN ANALYZE) — вау-деталь, но повседневной ценности мало; опционально позже.

## Стек

- **Движок**: DuckDB-WASM 1.5.4 (Web Worker, Arrow).
- **Сборка/деплой**: Vite + GitHub Pages (по шаблону stat·plan).
- **UI-каркас**: *открытый вопрос, решать первым в Claude Code.* Табы + перестановка блоков + состояние виджетов + несколько редакторов — это уже заметный компонентный стейт. Vanilla тянется, но React тут оправданнее, чем в скретчпаде. Решение влияет на грид/чарты ниже.
- **SQL-редактор**: CodeMirror 6.
- **Markdown**: лёгкий парсер (marked) для текстовых блоков.
- **Грид результата**: виртуализированный.
- **Визуализация**: Observable Plot (vanilla + Arrow) либо Chart.js; Recharts если React.
- **Экспорт**: HTML — сериализация отрендеренного отчёта; PDF — print-CSS + `window.print` (дёшево) или html2pdf/jsPDF (контроль над версткой).

## Риски и открытые вопросы

- **Данные испаряются при reload** — главная содержательная развилка. Reset — это намеренная очистка; reload — ненамеренная потеря. v1: «сохранить отчёт» сохраняет структуру, при открытии просим подгрузить датасеты заново. Элегантный апгрейд — OPFS (local-first, по твоей теме ODIN), в v1.5.
- **Layout creep** — как только блоки двигаются, тянет к «два в ряд» и обратно в грид. Firewall «вертикаль, не грид» держим жёстко.
- **Framework (vanilla vs React)** — табы + ноутбук-стейт усиливают аргумент за React; решить на старте.
- **Bundle size / холодный старт** — остаётся; честный лоадер.
- **Имя** — quackbook (рабочее).

## Фазы

- **v1** — слой данных (схема в рейле + Профиль + SQL с join/union) + табы исследования + Reset + ноутбук (текст/виджет-блоки с SQL и именем датасета, вертикальная перестановка) + сохранение структуры + экспорт HTML/PDF. Только локальные файлы.
- **v1.5** — OPFS-персист данных (отчёт открывается с данными, без релоада), key-хинт по джойнам, таб «План».
- **later** (firewall) — визуальный join-builder, dashboard-грид/канвас, правка значений, коллаборация.
