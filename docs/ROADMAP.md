# warpbook — роадмап после v1

Принят 2026-07-05 по итогам брейншторма (10 идей → выбор пользователя). Источник истины по v1 — `docs/scope-quackbook-v1.md`; этот документ описывает очередь развития **после** v1 и в части явно снятых ниже firewall-пунктов имеет приоритет над списком в CLAUDE.md.

## Зафиксированные решения

- В очередь взяты: витрина датасетов, REPL-углубление, исполняемые ячейки (M7b), автопрофиль, OPFS-персист, PWA, шаринг отчёта ссылкой, remote Parquet по allowlist, EN-локализация.
- **Снятие firewall (явный запрос 2026-07-05):** OPFS-персист (M11), шаринг ссылкой без сервера (M12), сетевое чтение Parquet по кураторскому allowlist (M13). Остальной firewall в силе: dashboard-грид/канвас, визуальный join-builder, per-cell правка значений и вычисляемые колонки в редакторе схемы, таб «План» (EXPLAIN).
- Не выбрано: ATTRACT MODE (автодемо-заставка) — вне очереди.
- **Решение 2026-07-06: NL→SQL снят с парковки** → майлстоун **M15**, идёт сразу после M10 (профиль — субстрат заземления). Состав: micro-RAG заземление на схему+профиль, NL-конструктор запроса (автокомплит сущностей в NL-вводе), WebLLM строго opt-in. Первая рантайм-зависимость `@mlc-ai/web-llm` **разрешена этим решением** (пункт «новые deps — только явным решением» соблюдён).
- Парковка (в самом конце, отдельное решение о входе): **светлая тема** (не дешёвая: хардкоды вне токенов — SQL-подсветка, plotFigure, свирл, вуаль/glow; путь: токен-гигиена → мокап → переключалка).

## Принцип порядка

Быстрые демо-вины → семантическое ядро → фичи поверх ядра → персист/оффлайн → наружу (шаринг, remote, охват).

Ключевая зависимость: **M7b меняет формат блока отчёта** (ячейка с редактируемым SQL и статусами), поэтому идёт раньше всего, что отчёт генерирует (M10 автопрофиль) или сериализует (M12 шаринг) — новый формат строится один раз, без миграций.

Вторая зависимость (2026-07-06): **M10 — субстрат M15**: NL→SQL заземляется на профиль значений; автопрофиль делает заземление всегда тёплым. Поэтому M15 сразу за M10. Номера майлстоунов — стабильные ID (ссылки в памяти/леджерах не ломаем), порядок задаёт таблица.

### M9 «Витрина + REPL»
- Обобщить welcome-экран в витрину датасетов: существующий cookbook-демо (payments + users) плюс 2–3 классических сэмпла (CSV и Parquet, суммарно ≤2–3 МБ в `public/`); клик → файлы зарегистрированы, схема в рейле.
- История запросов ↑/↓ в SQL-редакторе (как shell; персист в localStorage) + dot-команды в духе DuckDB CLI: `.tables`, `.schema <t>`, `.help` — перехват на клиенте до движка.
- DoD: визитёр без своего файла делает первый запрос < 30 сек; история переживает перезагрузку; `.schema demo_users` печатает колонки.
- TDD: ring-buffer истории, парсер dot-команд. Витрина — глазами.

### M7b «Исполняемые ячейки»
- Блок-виджет становится ячейкой: SQL редактируется прямо в отчёте и пере-исполняется; статусы (актуален / устарел / ошибка); «выполнить всё» по отчёту; блок «+ запрос» прямо в отчёте (в M4 сознательно не делали — теперь пора).
- Меняет сериализацию отчёта → идёт раньше M10 и M12.
- DoD: поменял данные → «выполнить всё» пересчитал весь отчёт; ячейка с ошибкой показывает её на месте, не ломая остальные.

### M10 «Автопрофиль»
- Кнопка на датасете → черновик отчёта из исполняемых ячеек: размер/null-карта, распределения числовых, top-K категориальных. Переиспользует профиль-код M3; генерация SQL — чистая логика под TDD.
- DoD: drop файла → «профилировать» → живой черновик за секунды, дальше редактируется как обычный отчёт.

### M15 «NL→SQL локально» (в очередь 2026-07-06, из парковки; идёт сразу после M10)
- **Заземление (micro-RAG):** чистая функция «схема + профиль → карточки колонок» (имя, тип, min/med/max, `approx_unique`, топ-K значений; ~50–100 токенов на колонку). Наш масштаб (1–4 таблицы) влезает в промпт целиком — retrieval/эмбеддинги НЕ нужны; при росте числа таблиц — лексический префильтр (слова вопроса ∩ имена/значения). Прямой субстрат — M10: профиль всегда тёплый. Карточки — TDD.
- **NL-конструктор (идея 2026-07-06):** автокомплит сущностей прямо в NL-вводе — упоминания таблиц/колонок/топ-значений из схемы+профиля тем же CM6-механизмом, что SQL-автокомплит. «выбери всех пин…» → подсказка `demo_penguins`; выбранное фиксируется каноническим токеном (чип/`@упоминание`) и уходит в промпт уже разрешённым. Синонимы/опечатки решаются ДО модели, детерминированно — это entity-linking в UI, вторая (после незнания домена значений) причина провалов text-to-SQL.
- **Модель:** WebLLM, строго opt-in — кнопка «включить ассистента (~1 ГБ)» с прогрессом (паттерн boot-прелоадера); кандидат Qwen2.5-Coder-1.5B q4, запасной 3B — выбор по eval'у. Нет WebGPU → фича скрыта с честным объяснением, приложение не деградирует.
- **Выход — черновик, не автозапуск:** сгенерированный SELECT пишется в активный таб (`-- вопрос: …` + SQL), юзер читает/правит/запускает. Guard: только одиночный SELECT (DDL/DML — отказ, наша инфра обёрток); repair-loop: одна повторная генерация с текстом ошибки DuckDB, дальше — руки юзера.
- **Приватность:** данные браузер не покидают (веса скачиваются К нам); проговорить в About.
- **Бенчмарк:** 10–15 канонических вопросов к сэмплам M9 (пингвины/такси/титаник) с ожидаемыми результатами; по нему выбор модели и go/no-go.
- DoD: «пингвины по видам» словами → рабочий SELECT в табе; ≥~70% бенчмарка верны с ≤1 repair; без WebGPU ничего не ломается.

### M11 «Инструмент»: OPFS + PWA
- OPFS: загруженные файлы и реестр переживают перезагрузку; квоты и «забыть файл»; спека начинается с проверки OPFS-поддержки в запиненном `@duckdb/duckdb-wasm` 1.32.0.
- PWA: manifest + рукописный service worker (0 deps), версионирование кэша под деплой (чтобы обновления не залипали), install prompt.
- DoD: закрыл вкладку → открыл оффлайн → данные и отчёт на месте, «выполнить всё» работает.

### M12 «Шаринг ссылкой»
- Отчёт + снапшоты результатов → gzip (`CompressionStream`, нативный) → base64url в `#fragment`; fragment не покидает браузер — данные не утекают на сервер.
- У получателя нет файлов: ячейки открываются замороженными (снапшот + SQL) с пометкой. Снапшоты — расширение формата (файл-отчёт результаты сознательно не сериализует); лимит полезной нагрузки и режим «без данных» — решить в спеке.
- DoD: ссылка открывается в чистом инкогнито и показывает отчёт; движок грузится лениво.

### M13 «Remote Parquet (витрина)»
- Кураторский allowlist источников с гарантированными CORS + range (HuggingFace datasets, R2 и т.п.) — секция в витрине M9. Произвольные URL остаются вне скоупа: хрупкость, из-за которой remote резался в v1, снимается курированием. Спека начинается с проверки https-range-чтения в запиненном 1.32.0.
- Индикатор «скачано X МБ из Y ГБ» как витринный вау-эффект.
- DoD: клик на датасет в десятки ГБ → агрегат за секунды, по сети — единицы МБ.

### M14 «EN-локализация»
- Своя строковая таблица (0 deps), переключалка RU/EN, персист выбора. Последней — чтобы волна по строкам была одна, после стабилизации UI.
- DoD: весь UI на обоих языках, включая витрину, `.help`, boot-экран и welcome.

## Процесс

Каждый майлстоун — полный цикл: брейншторм → спека (`docs/superpowers/specs/`) → план (`docs/superpowers/plans/`) → SDD. Гейт неизменен: `npm test` + `npm run build` + `npm run lint` 0/0; новые npm-зависимости — только отдельным явным решением. Миноры из `docs/BACKLOG.md` подшиваются к тематически ближайшему майлстоуну.
