# M10 «Автопрофиль» — дизайн

Дата: 2026-07-06. Статус: принят (брейншторм 2026-07-06). Источник очереди: `docs/ROADMAP.md` (M10).

## Зачем

Профиль M3 — эфемерная вьюха: не сохраняется, не экспортируется, всё зашито (k=7, bins=12). Автопрофиль превращает ту же информацию в **артефакт**: редактируемый отчёт из исполняемых ячеек (M7b), который экспортируется в HTML/PDF сейчас и шарится ссылкой в M12. Три ценности: (1) «глянул файл → отдал коллеге EDA-сводку»; (2) решение cold-start пустого отчёта — каждый график черновика это живой SQL, из которого начинается свой анализ; (3) демо-вин «EDA за клик» для визитёра без SQL. Связь «M10 — субстрат M15» из ROADMAP признана мнимой (M15 читает кэш профиля M3 напрямую) — ценность M10 самостоятельная.

## Решения брейншторма

1. **Append, не replace:** черновик дописывается в конец существующего отчёта; пустой отчёт просто наполняется. Ничего не теряется, конфирмов нет.
2. **Кап + отбор:** обзор и null-карта — всегда; пер-колоночных ячеек ≤ капа, отбор по eligibility из статистики профиля, порядок — как в схеме.
3. **Точка входа — рейл:** кнопка «в отчёт» рядом с «ПРОФИЛЬ ИСТОЧНИКА» для выбранного источника. Витрины (view/table) — вне этого захода.
4. **Генерация из схемы + статистики профиля M3** (подход 1): кэш `Dataset.profile` даёт честную классификацию (`ColumnKind`), eligibility и rowCount; нет кэша — считаем тем же путём, что кнопка профиля.

## UX-поток

Клик «в отчёт» → `profile(dataset)` при отсутствии кэша (busy-состояние на кнопке, как у профиля) → `buildProfileDraft(...)` → `appendBlocks(draft)` → `setMode('report')` → скролл к первому добавленному блоку → тост «профиль {fileName} добавлен: N ячеек». Ячейки M7b исполняются сами при монтировании — отдельный runAll не нужен; «живой черновик за секунды» получается из существующей механики.

## Состав черновика (порядок блоков)

Для датасета `t` (fileName `f`, R строк, C колонок):

1. **text:** `## Профиль: {f}` + строка `{t} · {R} строк · {C} колонок`.
2. **widget «null-карта»** (viz `table`), одна на таблицу, long-форма c guard от деления на 0:

```sql
SELECT * FROM (
  SELECT 'species' AS "колонка",
         count(*) FILTER (WHERE "species" IS NULL) AS "null",
         round(100.0 * count(*) FILTER (WHERE "species" IS NULL) / greatest(count(*), 1), 1) AS "%"
  FROM demo_penguins
  UNION ALL
  SELECT 'island', count(*) FILTER (WHERE "island" IS NULL),
         round(100.0 * count(*) FILTER (WHERE "island" IS NULL) / greatest(count(*), 1), 1)
  FROM demo_penguins
  -- ... по всем колонкам схемы
) ORDER BY "null" DESC, "колонка"
```

3. **По отобранным колонкам (≤ кап), в порядке схемы:**
   - numeric → **ячейка-гистограмма** (viz `chart`), самодостаточный SQL с CTE — min/max НЕ зашиты константами, ячейка переживает смену данных:

```sql
WITH s AS (
  SELECT min("body_mass_g") AS lo, max("body_mass_g") AS hi
  FROM demo_penguins WHERE "body_mass_g" IS NOT NULL
)
SELECT round(s.lo + floor(least(("body_mass_g" - s.lo) / nullif(s.hi - s.lo, 0) * 12, 11)) * (s.hi - s.lo) / 12, 2)::VARCHAR AS "от",
       count(*) AS "строк"
FROM demo_penguins, s
WHERE "body_mass_g" IS NOT NULL
GROUP BY 1 ORDER BY min("body_mass_g")
```

   (12 = `HISTOGRAM_BINS` из M3; `::VARCHAR` — чтобы buildChartSpec выбрал бакет осью X; порядок берёт `ORDER BY min(col)` — числовой, не лексикографический.)
   - categorical → **ячейка top-K** (viz `chart`):

```sql
SELECT "species" AS "значение", count(*) AS "строк"
FROM demo_penguins
WHERE "species" IS NOT NULL
GROUP BY 1 ORDER BY 2 DESC LIMIT 7
```

   (7 = `TOP_K` из M3.)

4. **text-хвост** (только если что-то не вошло): `Без своей ячейки остались: col (причина), …` — колонки за капом и неэлигибельные (high-cardinality, all-null, даты). Добавить руками легко — «+ запрос». *(Поправка 2026-07-06: формулировка выровнена по шипнутой — исходно «Не вошли в черновик»; смысл тот же, точнее отражает, что null-карта эти колонки всё равно покрывает.)*

Заголовки ячеек: `null-карта`, `{col} — распределение`, `{col} — топ значений`. `datasetNames: [t]` у каждой ячейки. Все имена колонок — через `quoteIdent` (core/sql).

## Кап и eligibility

- `PROFILE_CELL_CAP = 8` — максимум пер-колоночных ячеек (константа в core/autoprofile.ts).
- Eligibility по `ColumnProfile` M3: numeric — `stats` есть и `min < max` (ненулевой размах, не all-null); categorical — `2 ≤ distinct ≤ THRESHOLD_DISTINCT`. `range` (даты/время) и `highCardinality` — неэлигибельны в v1 (в хвост).
- Отбор: первые ≤ 8 элигибельных в порядке схемы. Детерминированно и тестируемо.

## Chart-слой: числовые строковые лейблы X

`plotFigure` сортирует bar по `-y` — гистограмма получила бы бары не в порядке бакетов. Мини-расширение по образцу `xDates`:

- `ChartSpec.xNumericStrings?: boolean` — ставится в `buildChartSpec`, когда sample-значение X — строка, парсящаяся в конечное число (`Number.isFinite(Number(v))`, не пустая).
- `plotFigure`: при `xNumericStrings` X-значения мапятся в `Number` и опция `sort` НЕ ставится — ординальный домен из чисел Plot сортирует по возрастанию → бары в порядке бакетов.
- Top-K ячейки не задеваются (лейблы не числовые → value-ranking как был). Регресс-тест на оба случая.

## Код

- **Новый `src/core/autoprofile.ts`** (чистый, TDD):
  - `PROFILE_CELL_CAP = 8`;
  - `type DraftBlock = Omit<TextBlock, 'id'> | Omit<WidgetBlock, 'id'>` (id минтит стор);
  - `buildNullMapSql(table, columnNames)`, `buildHistogramCellSql(table, col)`, `buildTopKSql(table, col)`;
  - `buildProfileDraft(input: { table: string; fileName: string; rowCount: number; columns: ColumnProfile[] }): DraftBlock[]` — состав, кап, eligibility, хвост.
- **Стор:** редьюсер `appendBlocks(blocks: DraftBlock[])` — минтит `blk-{seq}` id, дописывает в конец `report.blocks`, поднимает seq. (Отчётный формат `core/report.ts` НЕ меняется — блоки обычные text/widget v1.)
- **`src/core/chartSpec.ts` + `src/components/plotFigure.ts`:** `xNumericStrings` (выше).
- **Rail:** кнопка «в отчёт» рядом с «ПРОФИЛЬ ИСТОЧНИКА» (та же видимость — выбранный источник); оркестрация: кэш профиля → генерация → append → режим отчёта → скролл (`document.getElementById(blockId)?.scrollIntoView()` по образцу существующей навигации) → тост.

## Ошибки и эджи

- 0 строк / все колонки неэлигибельны → заголовок + null-карта + хвост со списком причин; гистограммы/top-K не генерятся.
- Профиль не посчитался (ошибка запроса) → тост с ошибкой, отчёт не трогаем (как ведёт себя кнопка профиля).
- Повторный клик по тому же датасету → второй черновик append'ится (осознанно: юзер сам решает, убрать ли старый; дедупликации нет).
- Кривые имена колонок (пробелы, кавычки, кириллица) → `quoteIdent` везде; лейбл колонки в null-карте — строковый литерал через `quoteLiteral` (core/sql, эскейп одинарных кавычек).

## Не берём (firewall + скоуп)

- Витрины (view/table) как источник автопрофиля — только файловые источники в этот заход.
- Временные распределения дат (`range`-колонки) — в хвост, не в ячейки.
- Дедупликация повторных черновиков, диалоги выбора колонок, настройка капа/бакетов в UI.
- Изменение формата отчёта (`core/report.ts`) — не требуется и не трогаем.
- Dashboard-грид, join-builder, per-cell правки — firewall как был.
