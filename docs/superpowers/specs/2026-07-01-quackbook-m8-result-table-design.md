# M8 «Интерактивная таблица результата» — Design

**Дата:** 2026-07-01 · **Ветка:** `m8-result-table` (от `main` `6404285`)

## Проблема

`run()` в Explore выполняет пользовательский SQL **как есть, без LIMIT**, а `arrowToRows` строит **JS-объекты на все строки** результата (`tab.result`). `SELECT * FROM huge` без лимита → миллионы JS-объектов в куче → вкладка виснет. Виртуализатор спасает только DOM, но весь массив строк всё равно в JS-памяти. Это латентный footgun, существующий уже сейчас.

Параллельно хочется дать интерактив как в itables/DataTables/Colab: сорт по клику, поиск, фильтр по колонке, пагинация. itables — Python-обёртка над DataTables.net (jQuery); нам не подходит (нет Python; jQuery + императивный полный DOM дерётся с React и де-виртуализирует грид). Но **у нас DuckDB-WASM прямо в браузере** — это лучший «сервер» для оконной подгрузки, чем фейковый сервер в ноутбуке.

## Решение (одной фразой)

На каждый SELECT **материализуем результат в снапшот-таблицу `_qb_result_<tab>`** (механизм уже есть — `buildResultTempDDL`/`resultTempName`, используется профилем), затем **листаем её страницами через `LIMIT/OFFSET`**, а сорт/поиск/фильтр **пушим в SQL** (`WHERE`/`ORDER BY`) по этой же таблице. В JS/DOM всегда только одно окно; вся масса — в колоночной WASM-куче DuckDB.

## Архитектура (Подход 1: всегда материализуем + пагинация)

**Жизненный цикл `run(sql)`:**
1. **Материализация:** `CREATE OR REPLACE TABLE _qb_result_<tab> AS <sql>`. Успех → режим `paged`. Провал (не-SELECT: PRAGMA/EXPLAIN/DDL руками) → фоллбэк `raw` (см. ниже). Ошибка запроса всплывает как сейчас (`setTabError`).
2. **Итог:** `SELECT count(*) FROM _qb_result_<tab>` → всего строк **M**; схема (колонки+типы) через `describeTable`.
3. **Первое окно:** `SELECT * FROM _qb_result_<tab> LIMIT size OFFSET 0` → грид рисует только его.

**Два режима результата** (дискриминант на табе):
- **`paged`** (основной) — есть пагинация / сорт / фильтр / профиль-по-таблице.
- **`raw`** (фоллбэк, редкий) — материализация не удалась → прямой `client.query(sql)`, показываем полный небольшой вывод как раньше, без пагинации/интерактива. Не регрессим.

**Состояние таба** (вместо «все строки в JS»):
- наличие результ-таблицы + `rowCount` (M) + `columns` (имена/типы);
- **`view`**: `{ page, pageSize, sorts: SortSpec[], search: string, filters: ColumnFilter[] }`;
- **`window`**: строки только текущей страницы (то, что рисует грид);
- `meta` (мс, всего строк);
- флаг загрузки окна + `mode: 'paged' | 'raw'`.

Смена `view` (страница/сорт/поиск/фильтр) → перезапрос окна из `_qb_result_` (эффект) → перерисовка страницы.

## Пагинация (классическая, страницы-кнопки)

Панель — **футер под гридом** (не в шапке — там pin/экспорт/витрина):
- **Размер страницы** — селектор `50 / 100 / 500` (дефолт **100**).
- **Навигация** — ⇤ ← → ⇥ (первая/пред/след/последняя), гасятся на границах.
- **Индикатор + прыжок** — «страница `[X]` из N» (X редактируемо, Enter → перейти) + «к строке `[ ]`» (прыжок на страницу с этой строкой). N = ⌈M / size⌉.
- **Диапазон** — «строки 101–200 из 12 480».

Поведение:
- Смена страницы → `… LIMIT size OFFSET (page−1)·size` с текущими `WHERE`/`ORDER BY`. OFFSET осознанно (детерминированный прыжок на страницу).
- Смена размера → сброс на стр. 1.
- При активном фильтре/поиске **M = отфильтрованный `count(*) … WHERE …`**; count пересчитываем при смене `WHERE`, не на сорт/страницу.
- Пусто (M=0) → «строк: 0», навигация выключена.
- `count(*)` приходит BigInt → `Number()` для арифметики. Прыжки clamp: страница [1, N], строка [1, M].
- **Гонка окон** — guard «последний выигрывает» (id/seq запроса), чтобы старый ответ не перезаписал новый вид.

## Сорт / поиск / фильтр (пушим в SQL, слой вида)

Всё меняет `WHERE`/`ORDER BY` по `_qb_result_<tab>`, живёт в `tab.view`, **редактор с исходным SQL не трогаем**.

- **Сорт** (клик по заголовку): tri-state нет → `ASC` → `DESC` → нет. **Shift+клик = мульти-сорт** (стрелка + номер приоритета). → `ORDER BY "colA" ASC, "colB" DESC`. Сброс на стр. 1, count не трогает.
- **Глобальный поиск** (строка в футере): `WHERE ("colA"::VARCHAR ILIKE '%q%' OR …)` по всем колонкам (числа/даты искабельны как текст). Экранируем значение + `%`/`_`. Дебаунс ~250 мс. Меняет `WHERE` → пересчёт count + стр. 1.
- **Фильтр по колонке** (воронка в заголовке → тип-зависимый поповер):
  - **Текст (VARCHAR):** contains / equals / starts-with (+ is null / not null).
  - **Число:** диапазон min–max (или `=`/`>`/`<`/between).
  - **Дата/таймстамп:** диапазон from–to.
  - **Низкая кардинальность** (distinct ≤ 50, определяем запросом при открытии поповера): чек-лист значений → `"col" IN (…)`.
- **Сборка `WHERE`:** `(глоб. поиск) AND (фильтр colA) AND (фильтр colB) …`. Экранирование: `quoteIdent` для колонок, `quoteLiteral` для значений.
- Активные фильтры — **чипсами** с «×» и «сбросить всё».
- **Copy-as-SQL:** кнопка эмитит переносимый запрос поверх **исходного** SQL (не внутреннее имя `_qb_result_`): `SELECT * FROM (<твой запрос>) WHERE … ORDER BY …` — вставить в редактор / запустить / запинить как виджет или витрину.

## График / профиль / пин / экспорт

- **График** (Observable Plot): на вкладке «график» — `SELECT * FROM _qb_result_<tab> [WHERE …] [ORDER BY …] LIMIT CHART_CAP` (~5000), **уважает активные фильтр/сорт**, но capped. Если отфильтрованных > CHART_CAP → плашка «график по первым N из M — агрегируй запросом». `buildChartSpec` не меняется, меняется только источник строк.
- **Профиль** — раз `run()` всегда материализует `_qb_result_<tab>`, профиль результата **больше не материализует сам** → мгновенный. Профилируем **полный** результат (без view-фильтров — это форма вывода запроса). Мелкий рефактор `useProfileActions`.
- **Пин / Витрина** — **без изменений** (работают с SQL из редактора). Отфильтрованный вид пинится/сохраняется через copy-as-SQL.
- **Экспорт (HTML/PDF)** — тот же footgun там есть (виджет с гигантским выводом раздует статический HTML). **Тем же capом ограничиваем таблицы в экспорте** (первые N + плашка «N из M»). Включено в веху.

## Крайние случаи / очистка

- **Ошибка запроса** — `CREATE TABLE AS <bad>` бросает → `setTabError`, грид скрыт; старые строки не показываем.
- **Ре-ран** (правка SQL + запуск) → новая материализация → **`view` сбрасывается** (сорт/фильтр/поиск сняты, стр. 1).
- **Загрузка окна** — лёгкий индикатор (притушить старую страницу / «загрузка…»).
- **Очистка** `_qb_result_<tab>`: на **закрытии таба** → `DROP TABLE _qb_result_<tabId>` (fire-and-forget). Ре-ран = `CREATE OR REPLACE` → на таб всегда одна таблица. Внутренние (`isInternalTable`) → не текут в рейл/автокомплит; умирают на перезагрузке.

## Тестируемое ядро (TDD-граница)

Чистый модуль **`core/resultQuery.ts`** — генерация SQL, покрывается red→green (Vitest node):
- `buildWindowSql(table, view, page, size)` → оконный SELECT с `WHERE`/`ORDER BY`/`LIMIT`/`OFFSET`.
- `buildCountSql(table, view)` → `SELECT count(*) … WHERE …`.
- `buildEffectiveSql(userSql, view)` → переносимый запрос для copy-as-SQL (обёртка исходного SQL).
- построители предикатов по типам (`buildOrderBy`, `buildWhere`, `buildGlobalSearch`, `buildColumnPredicate`) + экранирование.
Плюс store-логика (`view`-редьюсеры, режимы) — тоже TDD. **UI поверх (грид-заголовки, футер пагинации, поповеры фильтров, чипсы) — глазами** (нет jsdom/RTL, как во всём проекте).

Опционально нужны node-интеграционные тесты (`resultQuery.integration.test.ts`) над реальным DuckDB: материализация → count → окно → сорт/фильтр меняют выборку.

## Жёсткие ограничения

- **0 новых депов.** Переиспользуем `@tanstack/react-virtual` (уже есть) для грида; **НЕ** тащим `@tanstack/react-table`, jQuery, DataTables. Пагинация/сорт/фильтр — свой код (пара сотен строк) + генерация SQL к DuckDB.
- **Детерминизм:** без `Math.random`/`Date.now`/`new Date`; id окон/запросов — через store-`seq`.
- **Firewall:** сорт/фильтр — **read-only слой вида** над результатом запроса; **не** правка значений per-cell, **не** деривативные колонки в редакторе схемы, **не** визуальный join-builder. В скоупе.
- **TDD-граница:** ядро/стор — red→green (Vitest node); презентация — глазами.
- **Гейт каждой задачи:** `npm run lint` 0 / `npm run build` (tsc) / `npm test`.
- Коммиты — bash here-doc, трейлер `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`.

## Вне скоупа (осознанно)

- Column reorder / resize / freeze — не сейчас.
- Экспорт **отфильтрованного вида** отдельной кнопкой — через copy-as-SQL → редактор → экспорт/пин (не отдельный путь).
- Keyset-пагинация (глубокие офсеты) — OFFSET достаточно для jump-to-page; keyset при необходимости позже.
- Правка значений в гриде — за firewall.
