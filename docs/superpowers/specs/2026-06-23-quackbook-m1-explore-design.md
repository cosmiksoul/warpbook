# quackbook — Дизайн вехи M1 «Исследование»

- **Дата:** 2026-06-23
- **Статус:** утверждён к реализации (брейншторминг завершён, переход к плану)
- **Источник скоупа:** `scope-quackbook-v1.md` + `docs/superpowers/specs/2026-06-22-quackbook-delivery-design.md` (дорожная карта)
- **Правила:** `CLAUDE.md`
- **Предшествует:** M0 (сквозной скелет) — задеплоен, смерджен в `main`.

## Контекст и цель

M1 — самая большая веха v1 (вес L). Превращает M0-скелет (drop одного CSV → сырые строки) в **настоящий цикл исследования**: несколько файлов в одной in-memory DuckDB, схема в рейле, табы редакторов, SQL с JOIN/UNION между файлами, результат в виртуализированном гриде и на авто-чарте, подсветка читаемых колонок. Режим Отчёт пока заглушка (реальный — M4).

Депт-фёрст: M1 наращивает один полный слой поверх скелета и остаётся задеплоенной и демонстрируемой.

## Принятые решения

1. **Табы — бленд.** Клик по датасету в рейле открывает/фокусирует таб, преднастроенный на `SELECT * FROM <table>`; кнопка «+» добавляет пустой таб-черновик (например под JOIN). Все табы свободно редактируются и закрываются. (Развилка скоупа «1 таб = 1 датасет» vs IDE-черновики разрешена в пользу гибрида.)

2. **Чарт — авто.** Без UI-настройки осей в M1: первая нечисловая колонка → X (категория), первая числовая → Y. Тип: `line` если тип X-колонки — дата/время, иначе `bar`. Если числовой колонки в результате нет — тогл «График» выключен (нечего строить). (В M1 CSV весь `VARCHAR` ⇒ его даты не распознаются как время ⇒ для CSV всегда `bar`; Parquet с датой → `line`.) Отвечает «базовому чарту» из дорожной карты. Конфигурируемые оси/тип — позже, по запросу.

3. **Редактор — raw CodeMirror 6** (не обёртка `@uiw/react-codemirror`), тонкий React-компонент (~80 строк). Причина: M1 всё равно требует кастомного `Mod-Enter`-кейбинда и (позже) schema-автокомплита — обёртка не экономит сложную часть, но тянет лишние зависимости и тот же баг контролируемого значения.

4. **Чарт-либа — Observable Plot** за компонентом `<Chart>` (императивный рендер в ref-div). Модель данных Plot = массив row-объектов = наши `rows`; bar↔line — смена одной марки. Recharts 3.8.1 — рабочий запасной (React 19 ок), смена дешёвая за `<Chart>`.

5. **Грид — `@tanstack/react-virtual`, виртуализация только строк.** Горизонтальный скролл по колонкам — нативный; фикс. высота строки; sticky-заголовок. Колоночная виртуализация отложена (нужна лишь при 100+ колонках).

6. **Интроспекция типов — `DESCRIBE <table>`** для рейла (даёт DuckDB-имена `VARCHAR`/`DATE`/`DOUBLE` как в мокапе). Для метаданных результата (колонки грида) — Arrow-схема результата.

7. **CSV в M1 остаётся `all_varchar`.** Типизация (`sniff_csv` + `TRY_CAST`) — это M2. Parquet показывает родные типы сразу.

## Скоуп-лок

**В M1:**
- **Shell:** топбар, тогл Исследование/Отчёт (Отчёт = заглушка «появится в M4»), пилюля `local`, **Reset** (drop таблиц + очистка сессии).
- **Rail:** список источников (мульти CSV+Parquet), схема выбранного источника (имя + DuckDB-тип) с **pruning-подсветкой** колонок, читаемых текущим запросом.
- **Explore:** бленд-табы (закрываемые), CM6-редактор на таб, кнопка Run + ⌘↵.
- **Панель результата:** виртуализированный грид + мета (строк · мс), тогл Таблица/График, **авто-чарт** (bar/line).
- Мульти-файл **JOIN/UNION** через SQL (нативно, общая БД).
- Сессионный стор (Zustand).

**Отложено (не строим в M1 — чтобы не плодить мёртвый UI; добавляется в свою веху):**
- Кнопки `типы` (M2); `профиль источника` и вкладка `профиль` (M3); `закрепить в отчёт` (M4); `сохранить` (M4/M5).
- Автокомплит SQL по схеме (nice-to-have).
- Реальный режим Отчёт (M4).

## Стейт (Zustand)

```ts
interface Dataset {
  table: string          // sanitized SQL-идентификатор
  fileName: string       // исходное имя файла
  bytes: number          // File.size, захваченный при загрузке
  kind: 'csv' | 'parquet'
  columns: { name: string; type: string }[]  // из DESCRIBE
}
interface Tab {
  id: string
  title: string
  datasetTable: string | null   // null для пустого черновика
  sql: string
  result: QueryResult | null
  meta: { ms: number; rows: number } | null
  error: string | null
}
interface Session {
  datasets: Dataset[]
  tabs: Tab[]
  activeTabId: string | null
  mode: 'explore' | 'report'
}
```

Действия: `addDataset`, `openOrFocusTab(table)` (бленд: фокус существующего датасет-таба либо новый), `openBlankTab`, `closeTab(id)`, `setActiveTab(id)`, `updateTabSql(id, sql)`, `setTabResult(id, result, meta)`, `setTabError(id, msg)`, `setMode(mode)`, `reset()`.

`id` табов — детерминированный счётчик в сторе (без `Math.random`/`Date.now`, чтобы тесты были воспроизводимы).

## Поток данных

drop N файлов → на каждый: `registerFile` + load (CSV `all_varchar` / Parquet нативно) + имя таблицы из имени файла (sanitize; коллизии → суффикс) + `describeTable` → `addDataset`. Рейл показывает источники + схему. Клик по датасету → `openOrFocusTab` (сид `SELECT * FROM t`). Правка в CM6 → `updateTabSql`. Run (⌘↵) → query с замером `performance.now()` → `arrowToRows` → `setTabResult`. На изменении/прогоне SQL → `detectUsedColumns(sql, schemas)` → подсветка рейла. **Reset** → `DROP TABLE` каждой таблицы + очистка стора → пустое drop-состояние.

## Выверенная тех-база (research + проверка по офиц. источникам)

Уверенность по всем пунктам — высокая.

- **CM6:** один `EditorView` на таб (проще; изолирует undo/selection). Критичные паттерны:
  - синк внешнего значения **только при расхождении** (`if (value === view.state.doc.toString()) return` перед `dispatch`) — иначе курсор прыгает на 0 (классический баг CM6+React);
  - Run-кейбинд `Mod-Enter` в `Prec.high(...)`, `run()` возвращает `true` (съесть событие, не вставлять перевод строки);
  - mount-effect с `[]` (не зависеть от `value`), колбэки через ref (иначе устаревают);
  - схема-автокомплит (отложено): `sql({schema})` фиксирует схему при создании → обновление только через `Compartment`, не мутацией.
- **Plot:** `Plot.plot()` возвращает detached-узел → `el.replaceChildren(fig)` + `fig.remove()` в cleanup (иначе утечка SVG). Тестируем построение `ChartSpec` (чистое), не отрендеренный SVG (node-env).
- **react-virtual:** `useFlushSync:false` под React 19 (убирает предупреждение про flushSync при скролле); заголовок и тело делят один источник ширин колонок и один скролл-контейнер; абсолютные строки через `translateY(vi.start)` внутри спейсера высотой `getTotalSize()`.
- **DuckDB-WASM 1.32.0:** `read_parquet('file')` (канонично, = `parquet_scan`); `registerFileBuffer(name, Uint8Array)` лишь монтирует виртуальный файл → нужен `CREATE TABLE AS SELECT` для стабильной таблицы; мульти-файл = мульти-таблица в одной БД; байты таблицы — из `File.size` (pragma в `:memory:` бесполезна); мс — `performance.now()` вокруг `query` (честная wall-clock, включая transfer; не «время движка»); **дубли имён колонок** из JOIN схлопываются в `Record` → дедупить в `arrowToRows`.

## Обработка ошибок (минимум, CLAUDE.md rule 2)

- SQL-ошибка из DuckDB → текст в панели результата (`tab.error`), не падаем.
- Сбой загрузки файла (битый CSV/Parquet) → сообщение по этому файлу, остальные грузятся.
- Холодный старт WASM → честный лоадер (есть из M0).
- Никакой спекулятивной обработки под несуществующие сценарии.

## Стратегия тестирования (Vitest, TDD)

Red-green-refactor для всей логики `core/` и `state/`.

- **core:** `tableNameFromFilename` (sanitize, коллизии, пустой результат); `buildLoadCsv`/`buildLoadParquet` (корректный SQL, экранирование); `detectUsedColumns` (`SELECT *` → все колонки таблиц в FROM; `t.col`; JOIN по 2 таблицам; неизвестный идентификатор игнорируется; одинаковое имя в двух таблицах); `arrowToRows` дедуп дублей имён.
- **db (smoke, Node):** грузим CSV + Parquet, JOIN между ними, `DESCRIBE` отдаёт типы.
- **state:** `openOrFocusTab` (фокус существующего vs новый), `openBlankTab`, `closeTab`, `updateTabSql`, `reset`, детерминированные id.
- **точечно RTL:** переключение таба сохраняет содержимое редактора; Run наполняет грид.
- Презентация (CSS, раскладка, SVG чарта, виртуализированный DOM) — глазами (честная граница CLAUDE.md; node-env не меряет layout).

## Порядок сборки — два внутренних среза

M1 большой; собираем двумя срезами, каждый задеплоен и демонстрируем. Точная нарезка на задачи — в плане (writing-plans).

- **Срез 1 — фундамент Explore:** зависимости + стор + расширение `db/`+`core/sql` + мульти-загрузка (CSV+Parquet) + Shell + Rail (источники + схема, без подсветки) + один CM6-редактор + Run + виртуализированный грид + мета. **Демо:** гружу 2 файла, пишу JOIN, вижу строки в гриде.
- **Срез 2 — табы, pruning, чарт:** бленд-табы (TabStrip, openOrFocus/blank/close) + `core/pruning` + подсветка рейла + тогл Таблица/График + авто-чарт (Plot). **Демо:** несколько табов, рейл подсвечивает used-колонки, переключаюсь на график.

## Готово, когда (критерий вехи)

Гружу 2 файла (CSV + Parquet), открываю табы, пишу запрос с `JOIN` между ними, вижу результат в гриде и на чарте, рейл подсвечивает читаемые колонки, Reset чистит всё; на задеплоенном Pages-URL всё работает, reload → пустое состояние.

## Расхождения со скоупом/мокапом (зафиксированы)

- **CSV-типы:** в M1 все колонки CSV — `VARCHAR` (типизация = M2). Смешанные типы в мокапе (DATE/DOUBLE) — пост-M2 состояние.
- **Кнопки будущих вех** (`типы`, `профиль источника`, `профиль`-вкладка, `закрепить в отчёт`, `сохранить`) в M1 **не рендерим** — добавим в их вехах, чтобы не было мёртвого UI.
- **Имя типов в рейле:** показываем DuckDB-имена (`VARCHAR`, `DATE`, …) через `DESCRIBE`, не Arrow-имена (`Utf8`, `Int32`).

## Открытые мелочи / риски

- Точный пин `zustand` — выверить на скаффолде среза 1.
- `detectUsedColumns` — намеренно эвристика (парс идентификаторов), не полный SQL-парсер: возможны ложные срабатывания на алиасах/строковых литералах. Допустимо для подсветки-подсказки; не используется для корректности запроса.
- Дубли имён колонок (JOIN `SELECT *`) — дедуп в `arrowToRows`; альтернатива (индексные строки) не берём в M1.
